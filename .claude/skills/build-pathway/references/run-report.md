# RUN_REPORT — Schema and Append Helper

**File path:** `docs/prompts/<category>/RUN_REPORT.md`

This file accumulates entries across all stage runs. Every stage exit (success or failure) appends a block. It is the single place a human looks to understand what failed and what action is needed.

---

## Per-stage append format

```
## Stage <N> — <YYYY-MM-DD HH:MM> — <success|failed>
- invocation: <stage handler filename>
- duration: <hh:mm:ss or "n/a">
- failures:
    - <slug>: class=<error_class>, attempts=<n>, error=<msg>
- artifacts:
    - <abs path>  (sha256: <hex>, size: <bytes>)
- publish_gate:  (stage 6 only)
    - unresolved_blocking_todos: <n>
    - unverified_high_risk_claims: <n>
    - all_sections_reviewed_at_current_hash: <true|false>
    - passed: <true|false>
- next_action: |
    /build-pathway <cat>                         # auto-resume
    /build-pathway <cat> proceed <manifest_hash>  # gate
    manual: rebuild vault X — delete <path> and rerun
```

**Rules:**
- Append, never overwrite. Entries accumulate oldest-first.
- If there were no failures, omit the `failures:` block entirely.
- If there were no artifacts (early failure), omit the `artifacts:` block.
- SHA256 must be computed via `shasum -a 256 <file>` or `sha256sum <file>` before writing.

---

## Helper: how to append a RUN_REPORT block

Each stage handler must append to `docs/prompts/<category>/RUN_REPORT.md` at exit. Steps:

1. Record stage start time at the beginning of the stage handler (first step).
2. At stage exit, compute duration.
3. Build the block text per the format above.
4. **Append** the block to `RUN_REPORT.md` (do not overwrite).
   - If the file does not exist, create it with header:
     ```
     # RUN_REPORT — <category>
     Generated by build-pathway. One block per stage invocation.
     ```
   - Then append the block.
5. If stage 3 and printing REVIEW GATE, append the REVIEW GATE block to RUN_REPORT as well (so the user has both the console prompt and the file).

## Stage 6 publish_gate failure format

When publish_gate checks fail, append a detailed block:

```
## Stage 6 — <datetime> — failed (publish_gate)
- publish_gate checks:
    - unresolved_blocking_todos: <n>  [FAIL if > 0]
    - non_primary_corrections_applied: <n>  [FAIL if > 0]
    - unverified_high_risk_claims: <n>  [FAIL if > 0]
    - all_sections_reviewed_at_current_hash: <true|false>  [FAIL if false]
- failing checks: <list which checks failed>
- guide_hash in state: <hash>
- reviewed_against_guide_hash per section:
    - 01-destinations: <hash or "none">
    - 02-documents: <hash or "none">
    ... (all 6)
- next_action: |
    Resolve TODO: human-review markers and rerun stage 5, then stage 6.
    Search: grep -n "TODO: human-review" docs/guides/<category>.md
```

## Source audit findings format (stage 5 Mode B)

When Mode B audits claim markers and finds unverified claims, add to the stage 5 RUN_REPORT block:

```
- source_audit:
    - claim_id=<slug>: url=<url>, reason=<404|login-wall|claim-not-found|tier=unverified>
    ... (one line per flagged claim)
```
